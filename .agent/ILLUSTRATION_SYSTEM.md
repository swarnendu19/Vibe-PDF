# ILLUSTRATION_SYSTEM.md — Editorial Illustration Specification

This document details the visual style, creation rules, and thematic color mapping constraints for illustrations generated by the AI Publishing Engine.

---

## 1. Purpose
The purpose of this document is to specify the design standards for document illustrations. It ensures that the Illustration Agent generates vector graphics that align with the document's selected theme and layout grids.

---

## 2. Goals
* Standardize on a clean, vector-only flat editorial design.
* Enforce dynamic color mappings using CSS custom properties.
* Define layout size constraints for hero, sidebar, and inline figures.
* Establish validation rules to strip unneeded code during compilation.

---

## 3. Non-Goals
* **No Raster Assets**: We will not support PNG, JPEG, or WebP files for editorial illustrations in V1.
* **No Manual Adjustment Coordinates**: Renders are positioned automatically by the grid engine.

---

## 4. Architecture
Illustrations are managed as dynamic vector coordinates within the page rendering sandbox:

```
  Illustration Agent ──► SVG Code Payload ──► CSS Variable Injections ──► Print Compile
  Outputs clean vector   Injected in page      bg-secondary,             FOGRA39 CMYK
                         AST node              accent-primary            applied
```

---

## 5. Responsibilities
* **Visual Style Verification**: Ensures generated illustrations match permitted styles (Minimalist Linear, Flat Flat Editorial, Monochrome Blueprint).
* **Color Compatibility Validation**: Confirms that illustration elements reference CSS color variables.
* **Asset Integration**: Maps illustrations to their target page columns.

---

## 6. Dependencies
* **Color Alignment**: Inherits variable schemes defined in [COLOR_SYSTEM.md](file:///.agent/COLOR_SYSTEM.md).
* **Layout Sizing**: References page bounds configured in [DESIGN_SYSTEM.md](file:///.agent/DESIGN_SYSTEM.md).

---

## 7. Constraints
* **Vector Only**: All illustrations must be inline SVG structures.
* **No Blurs/Gradients**: Drop-shadow filters and radial blur coordinates are banned; depth must be represented using flat offset layers.
* **No External Links**: Illustrations must not contain image tags referencing external links.

---

## 8. Naming Conventions

### Illustration Attributes
* CSS class hooks: `api-illustration`, `api-illustration-hero`, `api-illustration-sidebar`.
* Element identifiers: `illust_[uuid]`.

---

## 9. Folder Structure
* Design guidelines and asset pools map to:
  * `/services/orchestrator/src/agents/illustrator.ts` (agent generation code).
  * `/apps/layout_sandbox/src/assets/illustrations/` (default fallback illustrations).

---

## 10. Design Decisions

### Banning Raster Formats
* **Decision**: We use only vector SVG files for illustrations, completely blocking PNG or JPEG uploads.
* **Rationale**: Raster formats pixelate when printed or scaled on high-resolution displays. SVGs scale cleanly, use less storage space, and can adapt to different themes dynamically.

---

## 11. Future Extensibility
* **Dynamic Animations**: The SVG format supports CSS transitions. If the engine expands to support interactive digital PDFs in the future, these illustrations can scale and animate without changing the underlying graphic structures.

---

## 12. Implementation Guidance
* Map SVG shape parameters to CSS variables.
* Run SVG payloads through optimization tools (such as SVGO) during compile steps.

---

## 13. Acceptance Criteria
* The layout app compiles successfully.
* Illustrations scale correctly to their column dimensions.
* Document compilation exports maintain color parity with the selected theme.

---

## 14. Common Mistakes
* **Hardcoded Backgrounds**: Specifying `#FFFFFF` as a background fill inside the SVG code, which stands out awkwardly when a dark theme is applied.
* **Complex Paths**: Generating overly complex vectors that take too long to compile and increase output file size.

---

## 15. Examples

### Semantic SVG Structure
```xml
<svg 
  xmlns="http://www.w3.org/2000/svg" 
  viewBox="0 0 500 200" 
  class="api-illustration api-illustration-hero"
>
  <!-- Background Shape -->
  <rect width="100%" height="100%" fill="var(--bg-secondary)" rx="4"/>
  
  <!-- Accent Graphics -->
  <circle cx="250" cy="100" r="50" fill="var(--accent-primary)"/>
  
  <!-- Narrative Lines -->
  <path d="M 50,100 L 450,100" stroke="var(--text-muted)" stroke-width="2"/>
</svg>
```

## 16. Decision Records

### ADR-010: Flat Design Depth Representation
* **Status**: Approved.
* **Context**: Complex gradient fills and CSS blur filters render poorly on standard office printers.
* **Decision**: Represent visual depth using flat offsets and solid fill overlays.
* **Consequence**: Ensures high-quality, high-contrast printing across all printer models.
